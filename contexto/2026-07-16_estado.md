## 0. Contexto crítico

Os PDFs de origem são **exportações "chapadas"**: páginas renderizadas inteiramente
como imagem, misturando **blocos de texto instrucional** e **prints de tela**. A
regra de negócio exige **texto nativo e editável no Word** (não imagem, não camada
oculta). A v3.0.0 mantém o pipeline de reconstrução do DOCX do zero, mas **troca a
análise de layout heurística (v2.0.0) por um modelo de Visão Computacional**.

## 1. Lógica / Código atual

Pipeline em 3 passos, por arquivo:

- **Passo 1 — Rasterização** (`conversor/rendering.py`): `pdf2image`/Poppler
  converte cada página em imagem BGR de alta resolução (300 dpi), uma por vez.
- **Passo 2 — Layout Analysis por ML** (`conversor/detector.py` + `layout.py` + `ocr.py`):
  - `detector.detect_regions()`: roda o **DocLayout-YOLO** na página e devolve os
    bounding boxes com as classes de layout, já mapeadas para **TEXTO / IMAGEM**;
  - `layout.analyze_page()`: para cada região TEXTO faz OCR no recorte, para IMAGEM
    guarda o bbox, e ordena os blocos no fluxo de leitura (top-to-bottom, esq→dir).
    Região de TEXTO com OCR vazio vira IMAGEM (preserva conteúdo);
  - `ocr.ocr_region_text()`: pré-processamento OpenCV (cinza + inversão de fundo
    escuro + Otsu) aplicado **só no recorte** de texto; OCR com PSM 6.
- **Passo 3 — Reconstrução** (`conversor/docx_builder.py`): `python-docx` remonta o
  DOCX na ordem de leitura, com parágrafos de texto nativo e imagens recortadas da
  **imagem original colorida**; quebra de página entre páginas.

### Modelo de IA

- **DocLayout-YOLO** — `juliozhao/DocLayout-YOLO-DocStructBench` (YOLOv10, pacote
  `doclayout_yolo`). Pesos **baixados automaticamente** do HuggingFace no 1º uso e
  cacheados.
- **Hardware (fallback automático):** `resolve_device()` usa `torch` para detectar
  GPU CUDA; havendo, roda nela, senão CPU. Registrado no log: `Hardware de inferência: ...`.
- **Mapeamento de classes → domínios** (`config.py`): `TEXT_CLASSES` (title,
  plain text, *_caption, table_footnote, …) → TEXTO; `IMAGE_CLASSES` (figure, table,
  isolate_formula, …) → IMAGEM; `DISCARD_CLASSES` (`abandon`) → descartada; classe
  desconhecida → IMAGEM (não perde conteúdo).
- **Multiprocessamento seguro:** pesos baixados **uma vez** no processo principal
  (`ensure_weights()`, sem corrida entre workers); modelo carregado **uma vez por
  worker** (singleton lazy `_MODEL` em `detector.py`), nunca por arquivo. Em GPU, o
  padrão é **1 worker** (evita OOM de VRAM).

### Arquitetura modular (pacote `conversor/`)

| Módulo | Responsabilidade |
| --- | --- |
| `config.py` | Dataclasses picláveis (`PipelineConfig`, `Block`, `ConversionResult`) + mapeamento de classes. |
| `dependencies.py` | Detecção de Poppler, idiomas do Tesseract e das dependências de IA (`ml_dependencies_available`). |
| `rendering.py` | Passo 1: rasterização das páginas. |
| `detector.py` | Passo 2 (IA): `resolve_device`, `ensure_weights`, singleton do modelo, `detect_regions`. |
| `ocr.py` | Pré-processamento + OCR por região (`ocr_region_text`) + `TESSDATA_PREFIX`. |
| `layout.py` | Passo 2: orquestra detecção + OCR das regiões + ordem de leitura (`analyze_page`). |
| `docx_builder.py` | Passo 3: reconstrução do DOCX (`DocxBuilder`). |
| `pipeline.py` | `convert_single` — orquestra os 3 passos (worker, fail-safe). |
| `main.py` | CLI, `discover_pdfs`, `_plan_jobs`, `run_batch`, resolução de hardware/pesos, `main`. |

### CLI

```powershell
python main.py                     # ./pdf -> ./docx (GPU se houver, senão CPU)
python main.py --dpi 400 --verbose
python main.py --ocr-lang eng
python main.py --workers 4 --force
python main.py --device cpu        # força CPU mesmo com GPU disponível
```

Flags: `--src`, `--dst`, `--workers`, `--force`, `--dpi` (300), `--ocr-lang`
(`por+eng`), `--min-conf` (40), `--device` (`auto`), `--yolo-conf` (0.2),
`--yolo-imgsz` (1024), `--verbose`. Exit code `1` se houver falha; `2` se Poppler,
Tesseract ou dependências de IA estiverem ausentes, ou se o download dos pesos falhar.

### Stack

- **Python 3.10–3.12 recomendado** (faixa com wheels estáveis de `torch`/`doclayout-yolo`;
  em 3.13/3.14 pode não haver wheel).
- Python deps (`requirements.txt`): `pdf2image==1.17.0`, `doclayout-yolo>=0.0.3`,
  `huggingface_hub>=0.24`, `torch`, `pytesseract==0.3.13`, `python-docx==1.2.0`,
  `opencv-python`, `numpy`, `Pillow`.
- **Binários de sistema:** Poppler (pdf2image) e Tesseract OCR (pytesseract),
  auto-detectados (`.poppler/`, `.tessdata/`).
- **GPU (opcional):** NVIDIA/CUDA acelera a inferência; instalar o wheel CUDA do
  PyTorch antes do `requirements.txt`.

## 2. Decisões técnicas firmadas nesta sessão

1. **DocLayout-YOLO em vez de `yolov8n.pt`**: o `yolov8n.pt` é treinado no COCO (80
   objetos do dia-a-dia) e **não detecta regiões de documento** — o mapeamento
   TEXTO/IMAGEM seria impossível. Escolhido o `DocStructBench` (YOLOv10 ajustado em
   documentos), que mantém intacto o resto do spec (ultralytics, download automático,
   fallback de hardware).
2. **Removida toda a heurística da v2.0.0**: projeção por bandas, `colorfulness`
   (Hasler), densidade de tinta, proporção alfabética, morfologia OpenCV e os limiares
   correspondentes (`color_threshold`, `min_words`, `fill_threshold`, etc.).
3. **Concorrência:** `ProcessPoolExecutor` (isolamento de falhas) com modelo carregado
   **1x por worker** (singleton lazy); pesos pré-baixados no processo principal; **GPU
   limitada a 1 worker** por padrão (VRAM).
4. **OCR por região**, não por página inteira; pré-processamento Otsu só no recorte de
   texto; imagem original colorida preservada para os prints.
5. **Preservação de conteúdo:** classe desconhecida e texto com OCR vazio viram IMAGEM.
6. **Mantidos:** pacote modular `conversor/`, fail-safe por processo, idempotência de
   nomes, filtro estrito `.pdf`, logging UTF-8, commits sem *Co-Authored-By*.
