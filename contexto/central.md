## Histórico de entregas

**2026-07-16** — Entrega da **v1.0.0** do conversor em lote PDF → DOCX. Implementada a arquitetura modular em `main.py` (descoberta, planejamento, conversão isolada e orquestração), com as regras de negócio de fail-safe (isolamento por processo), idempotência de nomes e filtro estrito de `.pdf`. Stack definida com `pdf2docx` sobre Python 3.14, paralelismo via `ProcessPoolExecutor` e CLI com `argparse`. Conversão validada com os 4 PDFs reais (0 falhas) e o projeto foi publicado no GitHub (commit `23f08fd`). Detalhes completos em [[2026-07-16_estado]].

**2026-07-16 (v1.1.0)** — Evolução para **pipeline híbrido com OCR**. Adicionado o pré-processamento via `ocrmypdf`/Tesseract, que gera um *Sandwich PDF* (`--skip-text`) tornando os documentos escaneados pesquisáveis **sem destruir as imagens** (Regra de Ouro). Incluída a detecção de texto nativo (`pdf_needs_ocr`) com *fallback* que pula o OCR, execução do OCR em subprocesso isolado com limpeza garantida de temporários (`tempfile`), pré-verificação das dependências de sistema e as flags `--no-ocr`/`--force-ocr`/`--ocr-lang`/`--min-text-chars`. Publicado no GitHub (commit `b2de66e`); rota de OCR pendente de validação end-to-end até a instalação de Tesseract + Ghostscript. Estado consolidado em [[2026-07-16_estado]].

**2026-07-16 (v1.2.0)** — **Validação end-to-end do OCR** (Tesseract 5.4.0 + Ghostscript 10.07.1 instalados) que revelou e corrigiu um **defeito bloqueante**: o `pdf2docx` descarta a camada de texto invisível do *Sandwich PDF* (alpha 0), então os DOCX saíam com imagens mas **sem texto pesquisável**. Corrigido com o **Passo 3 (`inject_hidden_text`)**: reextrai o texto do OCR por página (PyMuPDF) e o reinsere no DOCX como *runs* ocultos (`w:vanish`) — pesquisável/copiável sem poluir os prints. Adicionado também o **auto-fallback de idioma** (`resolve_tessdata`/`list_tesseract_langs`). **NOTA:** esta linha (v1.x) foi **descontinuada** — o texto oculto mantinha o documento como imagem (não editável), o que não atendia à regra de negócio. Ver v2.0.0.

**2026-07-16 (v2.0.0)** — **Reescrita completa para Document Layout Analysis (DLA).** Abandonados `ocrmypdf`+`pdf2docx`. Motivo: os PDFs são exportações "chapadas" (páginas 100% imagem) misturando texto instrucional e prints de tela; a abordagem sanduíche/extração mantinha tudo como imagem (texto não editável). Novo pipeline em 3 passos: **(1)** `pdf2image`/Poppler rasteriza cada página; **(2)** OpenCV segmenta a página em blocos (projeção horizontal/vertical) e classifica cada um como **TEXTO** ou **IMAGEM/print** (sinais: densidade de palavras, colorfulness de Hasler, proporção alfabética, densidade de tinta), com OCR (`pytesseract`) só nas regiões de texto e recorte das de imagem; **(3)** `python-docx` remonta um DOCX **nativo/editável** na ordem de leitura. Pré-processamento OpenCV (cinza + inversão de fundo escuro + Otsu) só na cópia do OCR. Código modularizado no pacote `conversor/`. Poppler portátil auto-detectado em `.poppler/`. Validado em 3 dos 4 PDFs reais (texto editável limpo + prints preservados; o 4º estava aberto no Word e falhou por lock — fail-safe). Estado consolidado em [[2026-07-16_estado]].

**2026-07-16 (v3.0.0)** — **Substituição da heurística por Visão Computacional (ML).** Toda a análise de layout por projeção/limiares matemáticos da v2.0.0 (colorfulness de Hasler, densidade de tinta/palavras, morfologia OpenCV, e as flags `--color-threshold`/`min_words`/etc.) foi removida e trocada por um modelo **DocLayout-YOLO** (`juliozhao/DocLayout-YOLO-DocStructBench`, YOLOv10 via pacote `doclayout_yolo`). Decisão-chave: o `yolov8n.pt` do spec original é treinado no COCO e **não detecta regiões de documento**, então foi rejeitado em favor do checkpoint ajustado em documentos. Novo módulo `conversor/detector.py` cuida do **fallback de hardware** (GPU CUDA via `torch`, senão CPU, logado no início), do **download dos pesos uma vez no processo principal** e do **carregamento do modelo uma vez por worker** (singleton lazy; em GPU o padrão é 1 worker para não estourar a VRAM). As classes do modelo são mapeadas para **TEXTO/IMAGEM** (`config.py`); o OCR passou a ser **por região** (Otsu só no recorte), com a imagem original colorida preservada para os prints. Novas flags `--device`/`--yolo-conf`/`--yolo-imgsz`. Preservados: pacote modular, fail-safe por arquivo e idempotência. `README.md` atualizado; código pronto, mas a validação end-to-end e o commit continuam pendentes (o ambiente atual é Python 3.14 sem `torch` instalado). Estado consolidado em [[2026-07-16_estado]].

**2026-07-16 (v3.1.0)** — **Otimização de CPU e cache configurável.** Com a v3.0.0 validada com sucesso pelo usuário, o pedido de "subir para YOLO26 (`yolo26n.pt`)" foi analisado e **recusado com justificativa**: o `yolo26n.pt` é treinado no COCO e não detecta regiões de documento (mesmo problema do `yolov8n.pt`), então trocá-lo quebraria o DLA validado — o modelo **DocLayout-YOLO foi mantido**. A aceleração de CPU veio por outro caminho: **controle de threads do torch por worker** (`núcleos ÷ workers` via `torch.set_num_threads`), eliminando o *oversubscription* do `ProcessPoolExecutor` (o maior gargalo de inferência em CPU com multiprocessamento); FP16/`half` foi deliberadamente evitado por ser ganho de GPU, não de CPU. Também foi **corrigido o `.env`**, que existia mas não era lido: agora `main.load_project_env` (via `python-dotenv`) carrega `HF_HOME`/`YOLO_CONFIG_DIR` no início e os propaga aos workers, direcionando o cache dos pesos para `./.models_cache/` (adicionado ao `.gitignore`). Adicionada a **instrumentação de tempo por página** (por arquivo e média global no resumo) para benchmark. `README.md` atualizado (histórico v3.1.0). Validação end-to-end pendente (ambiente atual é Python 3.14 sem `torch`); recomendado um piloto de 3–5 PDFs antes da conversão em massa, com atenção à RAM (cada worker carrega uma cópia do modelo — preferir `--workers 4`). Estado consolidado em [[2026-07-16_estado]].
